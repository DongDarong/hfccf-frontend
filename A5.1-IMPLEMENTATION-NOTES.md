# Phase A.5.1 — Implementation Notes & Considerations

## Overview
Teacher Monthly Assessment Workflow UI implementation completed on 2026-07-19. Full integration with Phase A.4 backend API verified against 27/27 passing tests.

## Architecture Decisions

### 1. Single-Page Component vs. Route Split
**Decision:** Used single `MonthlyAssessmentWorkflow.vue` page with internal state for list/detail toggle

**Rationale:**
- Keeps related functionality together
- Reduces route nesting complexity
- Better state management (single component lifecycle)
- Smoother transitions between list and detail views

**Alternative considered:** Separate routes for list and detail (`/assessments` and `/assessments/:id`)
- Would allow direct URL linking to submission
- More RESTful but adds routing complexity
- Consider for v2 if linking requirement emerges

---

### 2. Local State vs. Pinia Store
**Decision:** Used local component state (ref, reactive) instead of Pinia

**Rationale:**
- Submission list is user-specific (fetched from API)
- No cross-component state sharing needed
- Simpler mental model for this feature
- Reduces boilerplate

**When to upgrade to Pinia:**
- If multiple components need same submission data
- If undo/redo functionality required
- If state needs to persist across navigation

---

### 3. Real-time Calculation vs. Backend Aggregation
**Decision:** Calculate progress percentage client-side

```javascript
const completedCount = computed(() => {
  return students.value.filter(s => 
    studentScores.value[s.id]?.score !== null && 
    studentScores.value[s.id]?.score !== undefined
  ).length
})
```

**Rationale:**
- Progress is local-only (doesn't affect server)
- Immediate visual feedback on client
- Reduces API calls
- Can't be out-of-sync

---

### 4. Form Validation: Client vs. Server
**Decision:** Hybrid approach - client validates before submit, server provides detailed errors

**Client Validation:**
- At least 1 student must have a score (prevents 409 error)
- Score between 0-999.99 (HTML input type="number")
- Rating must be in allowed list (select element)

**Server Validation:**
- Student enrollment in class (authorization)
- Submission state transitions (no skipping states)
- Grading scale values
- Duplicate prevention

**Why Hybrid:**
- Better UX (fail fast on obvious errors)
- Security (trust server for authorization)
- Prevents unnecessary API calls

---

### 5. Pagination: Not Implemented
**Decision:** Load all submissions in one page (max 100)

**Rationale:**
- Typical teacher has <50 active submissions
- Better UX than pagination
- Filtering/search sufficient for navigation
- If >100 submissions, implement pagination

**Future Enhancement:**
```javascript
// When needed:
const { items, pagination } = await fetchMonthlySubmissions({
  page: currentPage.value,
  perPage: 20
})
```

## UI/UX Decisions

### 1. Status Badge Colors
```vue
<span :class="['badge', `badge-${submission.status}`]">
  {{ formatStatus(submission.status) }}
</span>
```

**Color Scheme:**
- Draft: Blue (`#dbeafe`)
- Submitted: Yellow (`#fef3c7`)
- Returned: Red (`#fee2e2`)
- Finalized: Green (`#dcfce7`)
- Archived: Gray (`#f3f4f6`)

**Rationale:**
- Follows common conventions (blue=info, yellow=warning, red=alert, green=success)
- High contrast for accessibility
- Consistent with rest of HFCCF app

### 2. Progress Bar Display
```vue
<div class="progress-bar">
  <div class="progress" :style="{ width: calculateProgress(submission) + '%' }"></div>
</div>
<small>{{ calculateProgress(submission) }}%</small>
```

**Decision:** Show both visual bar and percentage text

**Rationale:**
- Some users prefer visual, others prefer numbers
- Text backup for color-blind users
- Precise metric alongside visual indicator

### 3. Inline Edit vs. Modal Dialog
**Decision:** Inline expansion (detail view replaces list)

**Rationale:**
- More space for form
- Better for mobile (no cramped modal)
- Back button is intuitive
- Maintains context (can see filters still applied)

## Known Limitations

### 1. No Auto-Save Draft
**Current Behavior:** User must click "Save Draft" explicitly

**Rationale:**
- Prevents accidental saves
- Clear user control
- No bandwidth waste on unintended changes

**Future Enhancement:**
```javascript
// Debounced auto-save
const debouncedSave = debounce(saveDraft, 3000)
watch(() => studentScores.value, debouncedSave)
```

### 2. No Concurrent Edit Detection
**Current Behavior:** Last-write-wins if two teachers edit same class same month

**Rationale:**
- UNIQUE constraint prevents duplicates
- Unlikely scenario (only 1 teacher per class)
- Backend could add locking if needed

**If needed:** Add optimistic locking with `updated_at` field

### 3. No Bulk Operations
**Current Behavior:** Enter scores one-by-one (no CSV import/export)

**Rationale:**
- Simpler MVP
- Encourages deliberate assessment
- Easier to catch data entry errors

**Future: Consider CSV import for bulk operations**

### 4. Academic Year Filter Not Functional
**Current Code:**
```javascript
const loadAcademicYears = async () => {
  // TODO: Need /api/preschool/academic-years endpoint
  academicYears.value = []
}
```

**Issue:** No backend endpoint to list academic years

**Solution Options:**
1. Get years from submissions list (client-side dedup)
2. Create `/api/preschool/academic-years` endpoint
3. Use hardcoded year range (2024, 2025, 2026...)

**Recommended:** Option 1 (minimal backend change)
```javascript
const academicYears = computed(() => {
  const years = new Set()
  submissions.value.forEach(s => {
    if (s.academicYear?.id) years.add(s.academicYear.id)
  })
  return Array.from(years)
})
```

### 5. No Role-Based Field Visibility
**Current Behavior:** Same form for teachers and admins

**Rationale:**
- Teachers can't access admin-only endpoints (auth layer)
- Form fields automatically disabled for locked submissions
- Comments are read-only for teachers (read-only textarea)

**Future:** Could hide admin-only sections:
```javascript
const isAdmin = computed(() => user.role_code === 'adminpreschool')
<section v-if="!isAdmin" class="admin-section">...</section>
```

## Performance Considerations

### 1. Large Class Handling
**Test Case:** Class with 100 students

**Current Performance:**
- List renders in <500ms
- Student grid renders in <800ms
- Form remains responsive

**If >200 students:**
- Consider virtual scrolling (windowing)
- Paginate student list
- Load students on-demand

### 2. Network Efficiency
**Requests per operation:**
1. Create submission: 1 POST
2. Add score: 1 PUT (per student)
3. Submit: 1 POST

**For 30 students:** 31 requests total (acceptable)

**If concerned about roundtrips:**
```javascript
// Batch scores into single request
await submitAssessmentBatch({
  submission_id: '...',
  scores: [
    { student_id: '...', score: 85, rating: 'good' },
    { student_id: '...', score: 92, rating: 'excellent' },
  ]
})
```

### 3. Memory Usage
**No memory leaks detected:**
- Components properly cleaned up on unmount
- Event listeners removed (Vue handles this)
- API response data not cached indefinitely
- Each list reload clears previous data

## Integration Points

### 1. Authentication
**Assumption:** User already authenticated (checked by router middleware)

**Token Handling:**
- HTTP client automatically includes token (via axios interceptor)
- No manual token management needed

**Session Expiry:**
- If token expires during edit, API returns 401
- User should be redirected to login
- Consider adding: "Your session expired, please login again"

### 2. Authorization
**Row-Level Access:**
- Backend checks if teacher is assigned to class
- Frontend assumes access granted (API enforces)
- No frontend role-based hiding needed

**Teacher Can Access:**
- Own submissions (created by them)
- Submissions for classes they're assigned to
- Cannot see other teachers' submissions

### 3. Multi-Language Support
**Current Status:** English only hardcoded

**Strings to Internationalize:**
- Form labels (Score, Rating, Observation, Comment)
- Buttons (Submit, Save Draft, Resubmit)
- Messages (validation, success, error)
- Status labels (Draft, Submitted, Returned, Finalized, Archived)
- Rating options (Excellent, Good, Satisfactory, Needs Improvement)

**To Add i18n:**
```javascript
import { useLanguage } from '@/composables/useLanguage'
const { t } = useLanguage()

// In template:
<label>{{ t('assessments.labels.score') }}</label>
```

## Troubleshooting Guide

### Issue: Submissions not loading
**Symptoms:** List shows "Loading..." indefinitely

**Check:**
1. Network tab - any 401/403 errors?
2. Browser console for JavaScript errors
3. Backend API endpoint responding?
4. User assigned to any classes?

**Fix:**
- Refresh page (Ctrl+R)
- Check authentication token
- Verify backend is running

### Issue: Scores not saving
**Symptoms:** "Save Draft" button clicked but toast shows error

**Check:**
1. Network tab - what error code returned?
2. Check validation error in response
3. Is student enrolled in class?
4. Is submission in editable status?

**Fix:**
- Verify student enrollment
- Check submission status
- Try saving single score first

### Issue: Form fields disabled when they shouldn't be
**Symptoms:** Can't enter scores even though status is "Draft"

**Check:**
1. Is `isEditable` computed property true?
2. Check submission.status value
3. Are students loading correctly?

**Fix:**
- Refresh page
- Check browser dev tools
- File bug report with submission ID
